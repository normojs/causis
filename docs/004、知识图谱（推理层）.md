## 模块四：知识图谱（推理层）设计方案 v2.0

### 1. 模块定位

知识图谱层是 Causis 从“标准化事实”到“可推理知识”的核心转化层。模块三输出的标准化事实以三元组形式存在，但尚未形成关联网络。模块四负责将这些离散事实构建为**实体互联、关系可追溯、规则可执行的时间感知属性图**，并对外提供统一的图查询、混合检索与分析能力。

与因果推理、实体消歧等高级分析任务的职责划分明确：

| 职责                           | 归属       |
| :----------------------------- | :--------- |
| 属性图存储、增量更新与版本演化 | **模块四** |
| 图遍历、路径查询与图算法       | **模块四** |
| 向量语义搜索与图-向量混合检索  | **模块四** |
| 因果规则的结构化存储与触发     | **模块四** |
| 复杂因果推理与反事实推演       | 模块七     |
| 模糊实体消歧与关系补全         | 模块五     |
| 冲突检测、消解与裁决           | 模块六     |

---

### 2. 核心设计原则

- **极简单引擎**：采用单嵌入式图数据库，避免多引擎同步与运维的复杂性，保证部署形态为单二进制文件，零外部依赖。
- **开放标准优先**：对外查询接口采用 ISO 标准 Cypher/GQL，避免供应商锁定，降低开发者和用户的学习成本。
- **深度溯源集成**：图上的每条边和每个属性，均携带指向原始证据的溯源字段，支撑上层因果推理的可解释性。
- **时间感知建模**：所有关系均包含有效时间区间，支持时态图查询和历史版本回溯，匹配 Causis 的版本化数据湖基座。
- **图-向量融合**：在同一个引擎内实现结构遍历与语义相似度搜索，为混合推理级联提供核心支撑。

### 3. 技术选型：LadybugDB

#### 3.1 候选方案回顾

| 数据库        | 核心语言 | 嵌入     | Cypher       | 向量   | 许可       | 推理       | 关键风险             |
| :------------ | :------- | :------- | :----------- | :----- | :--------- | :--------- | :------------------- |
| **LadybugDB** | C++      | ✅ 嵌入式 | ✅ 原生       | ✅ 融合 | MIT        | 应用层实现 | —                    |
| CozoDB        | Rust     | ✅        | ❌ (Datalog)  | ✅      | MPL-2.0    | 强         | 开发停滞，维护风险高 |
| Samyama       | Rust     | 独立服务 | ✅            | ✅      | Apache 2.0 | 有限       | 学术主导，社区较小   |
| FalkorDB      | C        | 独立服务 | ✅            | ❌ 外部 | SSPLv1     | 有限       | 许可证风险高         |
| TypeDB 3      | Rust     | 服务部署 | ❌ (TypeQL)   | 规划中 | 社区版免费 | 最强       | 学习成本与部署复杂   |
| Grafeo        | Rust     | ✅        | ✅            | ✅      | MIT        | 有限       | 社区极小，维护风险   |
| OverGraph     | Rust     | ✅        | ✅ (类Cypher) | ✅      | MIT        | 有限       | 文档与生态尚不成熟   |

#### 3.2 最终选型：LadybugDB

**选择理由**：

1. **许可证友好**：MIT 许可，无 AGPL/SSPL 传染性风险，商业部署零顾虑。
2. **嵌入式架构**：类似 SQLite，单二进制部署，无需 Docker、无需独立服务，完美契合 Causis 的私有化交付和低运维目标。
3. **Cypher + SQL 双标准**：原生支持 Cypher 图查询和 SQL 关系查询，与模块三中的 DuckDB 生态无缝衔接——原有冲突检测 SQL 可直接在 LadybugDB 上执行。
4. **图-向量融合**：内置 HNSW 向量索引和全文搜索，支持一次查询中融合结构遍历与语义搜索，无需外挂向量数据库。
5. **多跳 OLAP 性能**：继承自 KuzuDB 的列式优化引擎，在 100K 节点 / 240 万边上执行 18 跳递归查询时性能远超行存图数据库。
6. **时间感知能力**：原生支持时间戳属性，可高效实现带时间窗口的边建模和时态图查询。
7. **成熟稳定**：由 KuzuDB 核心团队维护，社区活跃，项目健康度高。

**关键设计**：

- **统一查询适配器**：封装 LadybugDB 的 Cypher/SQL 双接口，为上层模块提供统一的 Rust trait 抽象，各模块无需感知底层是 Cypher 还是 SQL。
- **单引擎双协议**：核心存储与检索均在 LadybugDB 内完成，无需额外的同步或数据搬运。
- **推理在应用层**：规则传递闭包、因果链推导等递归推理逻辑，利用 LadybugDB 的多跳 OLAP 能力，在应用层自定义算法实现，架构保持简洁。

### 5. 核心功能设计

#### 5.1 从事实表到图谱的自动构建

- **事实→三元组转换器**：通过定时或增量触发管道，从 Apache Iceberg 事实表中读取 `entity_id`-`attribute`-`value` 三元组，自动生成图节点（实体）和属性边。
- **时间感知边建模**：利用 Iceberg 中的 `valid_from` / `valid_until` 字段，为每条边附加有效时间区间，实现时间窗口查询（例如“查询2023年张三的直属上级”）。
- **增量更新**：事实表新版本快照生成后，通过 Iceberg 快照差异对比，仅更新变更的实体和关系子图，避免全量重建。
- **溯源字段映射**：事实表中的 `_source_file`、`_provenance_id` 等字段同步写入图节点/边的属性，确保证据链不中断。

#### 5.2 因果规则与业务规则的结构化存储

- **规则节点建模**：将业务规则（如“请假需二级审批”）建模为图中的规则节点，与涉及的角色、部门、条件等实体用关系边连接。
- **规则触发**：当图上某实体状态变更（如某审批人状态变为“出差”），可触发应用层遍历规则子图，计算需要重新评估的因果链。
- **规则版本化**：规则本身作为节点，具有有效期属性，支持时间旅行查询，与公司制度文件的版本保持一致。

#### 5.3 图-向量混合检索

- **统一索引**：LadybugDB 对同一实体同时维护图邻接索引和 HNSW 向量索引，支持单查询中融合结构遍历与语义搜索。
- **典型查询**：“找到与‘深度学习框架’语义相关且和公司A有合作关系的团队”。引擎先通过向量相似度定位候选实体，再通过图关系精确过滤。
- **可解释性**：所有查询结果附带完整的图路径，结合溯源字段可直接展示证据链。

#### 5.4 图算法支持

- **内置路径算法**：最短路径、BFS/DFS、Dijkstra、PageRank、WCC（弱连通分量）。
- **因果路径搜索**：在应用层实现 `FindCausalPath`，利用 LadybugDB 的多跳遍历能力，在带 `causal:true` 标记的边上执行定向搜索，返回因果链路径及每步的置信度。
- **子图分析**：中心度、社区发现（Louvain）、共同邻居等，支持异常模式识别。

#### 5.5 时态图查询与版本回溯

- **时间点查询**：`MATCH (e:Employee {id:'E001'})-[r:ReportsTo]-(m) WHERE r.valid_from <= timestamp('2023-06-01') AND r.valid_until > timestamp('2023-06-01') RETURN m`。
- **版本对比**：同一关系的两个时间窗口快照，可直接通过 Cypher 做差集查询，识别审批链变化。

---

### 6. 查询语言标准：Cypher / SQL

- **开放标准**：Cypher 是事实上的图查询标准，GQL (ISO 39075) 已将其纳入国际规范，LadybugDB 的原生 Cypher 支持保证了生态兼容性。
- **扩展能力**：LadybugDB 同时支持 SQL，下游模块（如冲突协调）可继续使用熟悉的 SQL 进行聚合分析和质量检查，无需额外工具。

### 7. 图模型与 Schema 设计示例

以请假审批场景为例：

```
CREATE NODE TABLE Employee (
    entity_id STRING PRIMARY KEY,
    name STRING,
    status STRING,          -- 在职/出差/离职等
    _source_file STRING,
    _provenance_id UUID
);

CREATE REL TABLE ReportsTo (
    FROM Employee TO Employee,
    valid_from TIMESTAMP,
    valid_until TIMESTAMP,
    _provenance_id UUID,
    causal BOOLEAN DEFAULT false
);

CREATE NODE TABLE LeaveRequest (
    request_id UUID PRIMARY KEY,
    leave_type STRING,      -- 年假/病假等
    status STRING,
    _provenance_id UUID
);

CREATE REL TABLE ApprovedBy (
    FROM Employee TO LeaveRequest,
    role STRING,            -- “一级审批人”、“二级审批人”
    assigned_time TIMESTAMP,
    _provenance_id UUID
);
```

| 接口                 | 方向 | 数据内容                                           | 格式                                        | 引擎                           |

### 10. 总结

本方案以 **LadybugDB 单一嵌入式图数据库** 为核心，实现了 Causis 知识图谱层的全部功能，核心优势在于：

- **极简**：单引擎，无同步，编译为单一二进制，零运维负担。
- **高性能**：多跳 OLAP 加速，同时支撑 Cypher 图查询与 SQL 分析。
- **开放标准**：全面兼容 Cypher / GQL / SQL，零供应商锁定。
- **推理可演化**：推理能力在应用层按需构建，可在成熟期无缝对接专用推理引擎，架构演进平滑。
- **原生溯源**：每条边都携带从模块三继承的溯源字段，确保因果推理的解释链路完整贯通。
