# 开发顺序与功能实现规划

本文档定义 Causis 的开发顺序、功能列表、每个功能的作用、开发架构归属和实现方式。目标是把已有愿景落到可以排期、可以测试、可以演示的工程计划。

核心原则：

- 先做一条纵向闭环，再补横向能力。
- 先做领域模型和本地适配器，再接重型基础设施。
- 先保证证据链和审计链正确，再追求智能增强。
- 先确定性逻辑，再 LLM/模型辅助。
- 每个功能都必须服务“可信决策”这个主线。

## 1. 总体开发顺序

Causis 的开发顺序分为六个阶段：

| 阶段 | 名称 | 目标 | 结果 |
| --- | --- | --- | --- |
| Phase 0 | 工程地基 | 建立 Rust workspace、领域模型、测试规范 | 项目可开发、可测试、可持续演进 |
| Phase 1 | MVP 纵向闭环 | 跑通一个请假审批可信决策 demo | 从输入到解释的完整链路可复现 |
| Phase 2 | 真实数据接入 | 接入文档、数据库和持久化存储 | 可处理真实业务数据 |
| Phase 3 | 图谱与冲突增强 | 上真实图谱、实体消歧、冲突裁决 | 可信知识网络成形 |
| Phase 4 | 因果能力增强 | 支持干预、反事实、根因定位 | 从解释过去进入推演未来 |
| Phase 5 | 平台与生态 | Studio、SDK、MCP、Cube、Marketplace | 可被团队、企业和生态使用 |

## 2. 开发架构总览

推荐采用六边形架构：Core 只定义领域模型、服务接口和业务流程，外部基础设施通过 adapter 接入。

```text
apps/
  causis-cli
  causis-server
  causis-studio
  causis-mcp-server

crates/
  causis-core
  causis-ingest
  causis-evidence
  causis-facts
  causis-graph
  causis-resolution
  causis-causal
  causis-explain
  causis-api

adapters/
  causis-adapter-local
  causis-adapter-duckdb
  causis-adapter-rustfs
  causis-adapter-iceberg
  causis-adapter-ladybug
  causis-adapter-cube
  causis-adapter-llm
```

依赖方向：

```text
apps -> api/cli -> core services -> ports/traits -> adapters
```

禁止方向：

```text
core -> concrete infrastructure
core -> Cube
core -> LLM provider
core -> LadybugDB concrete client
core -> RustFS concrete client
```

## 3. Phase 0：工程地基

### 3.1 初始化 Rust Workspace

**作用**

建立项目工程骨架，让 Causis 从文档项目变成可开发工程。

**开发架构**

- 归属：repo root。
- 产物：`Cargo.toml` workspace、`crates/`、`apps/`、`adapters/`。
- 初始最小 crate：`causis-core`、`causis-cli`。

**实现方式**

创建 workspace，不急于创建所有 crate。第一步只创建能承载领域模型和 demo runner 的最小结构。`causis-core` 存放 ID、错误、配置、领域类型；`causis-cli` 提供本地命令入口。后续按功能压力再拆分其他 crate。

**验收标准**

- `cargo test --workspace` 可运行。
- `cargo fmt --all --check` 可运行。
- workspace 不依赖任何外部服务。

### 3.2 定义核心 ID 与领域类型

**作用**

建立 Causis 的可信链路基础。所有证据、事实、冲突、推理、解释都必须有稳定 ID。

**开发架构**

- 归属：`crates/causis-core`。
- 关键类型：`EvidenceId`、`ProvenanceId`、`EntityId`、`FactId`、`ConflictGroupId`、`TraceId`、`DecisionId`。

**实现方式**

使用 newtype 包装 UUID 或 ULID，避免到处传裸字符串。所有 ID 实现 `Serialize`、`Deserialize`、`Display`、`FromStr`。为了测试稳定，可以允许 fixture 使用固定 ID。

**验收标准**

- 每个 ID 有单元测试。
- JSON 序列化稳定。
- 错误信息可读。

### 3.3 定义 Provenance Schema

**作用**

保证任何派生事实都能追溯到原始证据。

**开发架构**

- 归属：`crates/causis-core`，后续可拆到 `causis-evidence`。
- 类型：`EvidenceAnchor`、`ProvenanceEvent`、`ProvenanceChain`。

**实现方式**

先定义纯 Rust 结构体和 JSON 表达，不接 Iceberg。`EvidenceAnchor` 表示文件位置，例如页码、段落、表格行列、字符范围。`ProvenanceEvent` 表示一次处理步骤，例如 ingest、parse、normalize、resolve、infer。`ProvenanceChain` 串起多个事件。

**验收标准**

- 能从一个 `Fact` 找到源文件位置。
- 能表示“文档第 3 页第 2 段”这种锚点。
- 能表示处理链：上传 -> 解析 -> 抽取 -> 标准化。

### 3.4 建立错误、配置和日志规范

**作用**

让项目早期就具备可维护性，避免每个模块各自定义错误和配置方式。

**开发架构**

- 归属：`crates/causis-core`。
- 技术：`thiserror`、`serde`、`tracing`。

**实现方式**

定义 `CausisError` 和 `Result<T>`。配置先采用 TOML/JSON 文件，不接复杂配置中心。日志统一使用 `tracing`，CLI 默认输出简洁日志，debug 模式输出详细链路。

**验收标准**

- 所有 public API 返回统一 Result。
- CLI 可以切换 log level。
- fixture 测试中错误快照稳定。

### 3.5 CI 基线

**作用**

保证后续每个功能都有自动检查。

**开发架构**

- 归属：`.github/workflows/ci.yml`。
- 检查：fmt、clippy、test。

**实现方式**

添加 GitHub Actions。早期不拉外部服务，不跑 Docker，不启动 RustFS 或数据库。等 adapter 集成后再增加集成测试 job。

**验收标准**

- PR 或 push 能自动检查。
- CI 时间保持短。
- 不依赖本地隐藏配置。

## 4. Phase 1：MVP 纵向闭环

Phase 1 的目标是先跑通一个可复现的 demo，而不是把所有模块都做完。

推荐场景：请假审批自动分发。

输入：

- 员工关系。
- 请假申请。
- 代理人规则。
- 审批制度片段。

输出：

- 最终审批人。
- 因果路径。
- 证据链。
- 审计记录。

### 4.1 Fixture 数据集

**作用**

提供稳定、可测试、可演示的数据基础。

**开发架构**

- 归属：`fixtures/leave-approval/`。
- 数据格式：JSON、CSV、Markdown，后续增加 PDF/DOCX/XLSX。

**实现方式**

先用结构化 JSON 表达制度、员工、审批单和状态，不急于解析真实 PDF。每条 fixture 都包含模拟证据锚点，例如 `policy.md#section-5.2`、`employees.csv#row-3`。

**验收标准**

- 一条 CLI 命令加载 fixture。
- fixture 不依赖外部服务。
- fixture 中至少包含一个冲突样例。

### 4.2 本地 Evidence Store

**作用**

保存证据元数据，并为后续 RustFS/S3 接入保留接口。

**开发架构**

- trait：`EvidenceStore`。
- P0 adapter：`LocalEvidenceStore`。
- 后续 adapter：`RustFsEvidenceStore`、`S3EvidenceStore`。

**实现方式**

P0 不复制大文件，只登记本地文件路径、hash、mime type、size、logical source 和 source version。写入本地 JSONL 或内存 map。接口返回 `EvidenceId` 和 `EvidenceRecord`。

**验收标准**

- 能登记 fixture 中的源文件。
- 能计算或记录 hash。
- 能按 `EvidenceId` 查询证据元数据。

### 4.3 Provenance Tracker

**作用**

记录数据从证据到事实再到推理结果的完整处理过程。

**开发架构**

- trait：`ProvenanceTracker`。
- P0 adapter：内存或 JSONL。
- 后续 adapter：Iceberg provenance table、LadybugDB audit node。

**实现方式**

每次处理动作追加一个 `ProvenanceEvent`。事件包括输入 ID、输出 ID、操作名称、时间、actor、工具版本和锚点。P0 中 actor 可以是 `system`。

**验收标准**

- 从 `TraceId` 能找到相关 `FactId`。
- 从 `FactId` 能找到 `ProvenanceId`。
- 从 `ProvenanceId` 能找到 `EvidenceAnchor`。

### 4.4 Fact Model 与 Fact Store

**作用**

把输入数据统一为标准事实，为图谱、冲突检测和因果推理提供基础。

**开发架构**

- crate：`causis-facts`。
- trait：`FactStore`。
- 类型：`Fact`、`FactValue`、`ValidityWindow`、`QualityScore`。

**实现方式**

P0 用内存实现。事实采用 entity-attribute-value 结构，并带 `valid_from`、`valid_until`、`provenance_id`、`quality_score`。例如：

```text
Employee:E001 reports_to Employee:E002
Employee:E002 status business_trip
Employee:E002 delegate Employee:E003
Policy:LeaveApproval rule fallback_to_delegate
```

**验收标准**

- fixture 能转成 facts。
- 每条 fact 都有 provenance。
- facts 能按 entity 和 attribute 查询。

### 4.5 基础标准化管线

**作用**

将不同来源的字段、枚举和时间格式转成统一表达。

**开发架构**

- crate：`causis-facts`。
- 组件：`Normalizer`。
- 配置：`normalization.toml` 或内置规则。

**实现方式**

P0 只做有限规则：日期标准化、人员 ID 统一、状态枚举映射、空值处理。保留 `value_raw` 和 `value_normalized`，不覆盖原始值。

**验收标准**

- 原始值和标准值同时存在。
- 错误值能产生 warning，不中断整个 demo。
- 标准化规则有单元测试。

### 4.6 L1/L2 实体对齐

**作用**

将来自不同数据源的同一实体归并到稳定 `EntityId`。

**开发架构**

- crate：`causis-facts` 或 `causis-graph` 前置组件。
- 组件：`EntityAligner`。

**实现方式**

P0 只做确定性对齐：员工工号、邮箱、制度编号、申请单号。L2 可以支持姓名 + 部门 + 日期组合匹配，但低置信度不自动合并，只打 `ambiguous` 标记。

**验收标准**

- 强 ID 一致时自动对齐。
- 模糊候选不误合并。
- 对齐决策有日志和 provenance。

### 4.7 冲突检测与标记

**作用**

发现多源事实之间的不一致，但不做最终裁决。

**开发架构**

- crate：`causis-facts`。
- 组件：`ConflictDetector`。
- 输出：`ConflictGroup`。

**实现方式**

按 `entity_id + attribute + valid_time` 分组，如果标准值不同，生成 `ConflictGroupId`。P0 支持精确冲突，例如同一员工当前主管不同、同一制度条款版本不同。

**验收标准**

- 检测出 fixture 中的冲突。
- 冲突事实都保留。
- 每条冲突 fact 标记 `conflict_status=detected`。

### 4.8 In-memory Graph Store

**作用**

把事实构建为可遍历图，为审批链和因果路径提供结构基础。

**开发架构**

- crate：`causis-graph`。
- trait：`GraphStore`。
- P0 adapter：`InMemoryGraphStore`。
- 后续 adapter：`LadybugGraphStore`。

**实现方式**

用节点和边结构表达员工、申请、制度、规则和状态。P0 支持基本查询：查直属上级、查代理人、查规则适用条件、查路径。

**验收标准**

- facts 能导入 graph。
- 能查询员工审批链。
- 图边带 provenance。

### 4.9 确定性冲突裁决

**作用**

在冲突影响审批链时，给出透明、可审计的采信结果。

**开发架构**

- crate：`causis-resolution`。
- trait：`ResolutionEngine`。
- 策略：authority first、latest valid time、quality score。

**实现方式**

P0 只做规则策略链，不接 LLM。配置每个 source 的权威分。例如 HR 系统高于手动表格。裁决结果写入 `ResolutionDecision`，保留 adopted value 和 rejected alternatives。

**验收标准**

- 冲突组能被裁决。
- 裁决理由可读。
- 未采纳版本保留。

### 4.10 Causal Trace 查询

**作用**

回答“为什么审批人是 X”，输出因果路径。

**开发架构**

- crate：`causis-causal`。
- trait：`CausalEngine`。
- P0 实现：规则图上的路径追溯，不做复杂 SCM。

**实现方式**

将审批决策拆成路径：

```text
LeaveRequest -> Applicant -> ReportsTo -> ManagerStatus -> Rule -> Delegate -> Approver
```

P0 的 causal trace 是确定性路径解释。每条边带置信度、规则名、provenance。P1/P2 再接 DeepCausality 做干预和反事实。

**验收标准**

- 输入申请单 ID，输出审批人。
- 输出完整路径。
- 路径每一步能找到证据。

### 4.11 Explanation Builder

**作用**

把 trace 转成用户和审计人员能理解的结构化解释。

**开发架构**

- crate：`causis-explain`。
- 组件：`ExplanationBuilder`。
- 输出：`ExplanationJson`。

**实现方式**

P0 输出 JSON，不做自然语言润色。结构包括 conclusion、steps、evidence、conflicts、resolution、limitations。README 或 demo 可以再把 JSON 渲染成 Markdown。

**验收标准**

- 每一步包含 source anchor。
- 冲突和裁决显示在解释里。
- JSON schema 稳定。

### 4.12 CLI Demo

**作用**

让项目能被一条命令演示，形成产品信心和开发闭环。

**开发架构**

- crate/app：`causis-cli`。
- 命令：`causis demo leave-approval`。

**实现方式**

CLI 读取 fixture，按顺序执行 evidence -> facts -> graph -> conflicts -> resolution -> trace -> explanation，然后输出 JSON 和简短摘要。

**验收标准**

- 一条命令跑通。
- 输出最终审批人。
- 输出证据链文件。
- 测试中可复现。

## 5. Phase 2：真实数据接入

### 5.1 Document IR

**作用**

为 PDF/DOCX/XLSX/PPTX 等文档解析提供统一中间表示。

**开发架构**

- crate：`causis-ingest`。
- trait：`DocumentParser`。
- 类型：`Document`、`DocumentBlock`、`TableBlock`、`TextSpan`、`BoundingBox`。

**实现方式**

先支持 Markdown/CSV/JSON，再支持 DOCX/XLSX，最后支持 PDF。每个 block 都生成可追溯 anchor。扫描版 OCR 放到后续增强。

**验收标准**

- 能从文档生成结构树。
- 表格单元格有 row/column anchor。
- 段落有 section/page anchor。

### 5.2 DocTor MVP

**作用**

实现 Causis 自有文档解析能力，减少对外部服务依赖。

**开发架构**

- crate：`causis-ingest`。
- adapter：`DocxParser`、`XlsxParser`、`PdfTextParser`。

**实现方式**

DOCX/PPTX 走 zip + XML 解析，XLSX 用 calamine，PDF 先做文本型 PDF。解析结果统一输出 Document IR。

**验收标准**

- 能解析制度文档。
- 能提取表格。
- 能保留来源位置。

### 5.3 SourceConnector

**作用**

统一文件、数据库、对象存储、流数据的接入方式。

**开发架构**

- crate：`causis-ingest`。
- trait：`SourceConnector`。
- P1 adapter：`LocalFileConnector`、`PostgresConnector`。

**实现方式**

定义 `connect()`、`discover_schema()`、`read_batch()`。批数据用 Arrow RecordBatch 表示。CDC 后续做，P1 先批量读。

**验收标准**

- 本地文件和 Postgres 数据能进入统一管线。
- 输出 schema 可检查。
- 错误能定位到具体 source。

### 5.4 DuckDB/Postgres Fact Store

**作用**

让事实层从内存进入真实查询能力，为 Cube 和 BI 做准备。

**开发架构**

- adapter：`causis-adapter-duckdb` 或 `causis-adapter-postgres`。
- trait：继续实现 `FactStore`。

**实现方式**

P1 推荐先 DuckDB 或 Postgres，避免一开始就接完整 Iceberg。事实表、冲突表、裁决表都可以用 SQL 表达。后续再迁移或同步到 Iceberg。

**验收标准**

- facts 可 SQL 查询。
- conflict groups 可 SQL 查询。
- trace 可通过 fact id 找回证据。

### 5.5 RustFS/S3 Evidence Adapter

**作用**

让证据湖具备真实对象存储能力。

**开发架构**

- adapter：`causis-adapter-rustfs`。
- trait：`EvidenceStore`。

**实现方式**

将原始文件写入 S3-compatible bucket，记录 object key、version、hash、size。P1 可先兼容通用 S3 API，不强依赖 RustFS 特性。

**验收标准**

- 文件可写入对象存储。
- 元数据可查询。
- hash 校验通过。

### 5.6 Iceberg Fact Store

**作用**

支持事实表快照、时间旅行和 schema 演进。

**开发架构**

- adapter：`causis-adapter-iceberg`。
- trait：`FactStore` 或 `SnapshotFactStore`。

**实现方式**

不要让 core 直接依赖 iceberg-rust。adapter 负责将 `Fact` 映射到 Iceberg schema。P1 可先写事实表和 provenance 表，P2 再做复杂 MERGE 和时间旅行查询。

**验收标准**

- 能写入事实快照。
- 能读取指定版本 facts。
- schema 变更有测试。

## 6. Phase 3：图谱与冲突增强

### 6.1 LadybugDB Graph Adapter

**作用**

将 in-memory graph 替换为真实图数据库能力。

**开发架构**

- adapter：`causis-adapter-ladybug`。
- trait：`GraphStore`。

**实现方式**

adapter 负责 Cypher/SQL 映射。Core 仍使用节点、边、路径、子图这些领域类型。先支持基础节点边写入、路径查询、时态边过滤，再支持向量检索。

**验收标准**

- 能写入员工、制度、规则图。
- 能查询审批链路径。
- 图边 provenance 保留。

### 6.2 Rule Graph

**作用**

把制度规则显式建模为可查询、可版本化的图节点。

**开发架构**

- crate：`causis-graph`。
- 类型：`RuleNode`、`ConditionEdge`、`ActionEdge`。

**实现方式**

将规则表达为条件和动作。例如“直属上级出差时，流转给代理人”。P1 可以用有限 DSL 或结构化 JSON，不急于做完整规则语言。

**验收标准**

- 规则可版本化。
- trace 能引用规则节点。
- 规则有来源条款 anchor。

### 6.3 语义实体消歧

**作用**

处理没有强 ID 的模糊实体。

**开发架构**

- crate：`causis-disambiguation` 或并入 `causis-graph` 的服务层。
- adapter：embedding model adapter。

**实现方式**

先生成候选集，再用字符串相似度、属性重合度、图邻域相似度评分。P1 后期再接 Candle/ONNX embedding。低置信度进入人工审核队列。

**验收标准**

- 候选排序可解释。
- 自动合并有阈值。
- 不确定项不自动合并。

### 6.4 高级冲突协调

**作用**

从确定性裁决升级到策略链和人在回路。

**开发架构**

- crate：`causis-resolution`。
- 类型：`ResolutionPolicy`、`ResolutionDecision`、`ReviewTask`。

**实现方式**

裁决策略做成可组合链：authority -> recency -> quality -> voting -> manual review。LLM 只生成建议和理由，不直接写最终结果。

**验收标准**

- 策略链可配置。
- 人工审核结果可回写。
- rejected alternatives 始终保留。

### 6.5 审核队列 API

**作用**

让人在回路真正可操作。

**开发架构**

- crate：`causis-api`。
- endpoint：`GET /review/tasks`、`POST /review/tasks/{id}/resolve`。

**实现方式**

API 返回冲突上下文、候选事实、证据锚点、系统建议。提交审核时生成 `ResolutionDecision` 和 audit event。

**验收标准**

- 可列出待审核项。
- 可提交人工裁决。
- 裁决进入解释链。

## 7. Phase 4：因果能力增强

### 7.1 Causal Graph Model

**作用**

把普通业务关系和因果关系区分开。

**开发架构**

- crate：`causis-causal`。
- 类型：`CausalNode`、`CausalEdge`、`CausalGraphVersion`。

**实现方式**

在图边上增加 causal semantics：direction、confidence、mechanism、assumption、evidence。不是所有关系都是因果边。

**验收标准**

- 能导出因果子图。
- 因果边有证据和假设。
- 非因果关系不会被用于因果推理。

### 7.2 DeepCausality Adapter

**作用**

引入形式化因果推理能力。

**开发架构**

- adapter：`causis-adapter-deepcausality`。
- trait：`CausalEngine`。

**实现方式**

先实现简单效应传播和 trace，再逐步支持干预。adapter 负责把 Causis causal graph 转成 DeepCausality 所需结构。

**验收标准**

- CausalEngine trait 不绑定具体库。
- 同一个测试可用 mock engine 和 real engine 跑。
- 推理结果可审计。

### 7.3 干预分析

**作用**

回答“如果设置 X，会不会影响 Y”。

**开发架构**

- crate：`causis-causal`。
- API：`POST /api/v1/query/intervention`。

**实现方式**

定义 `InterventionQuery`，包含 target、intervention variable、new value、graph version。输出 effect estimate、affected path、assumptions、limitations。

**验收标准**

- 输出不只给结论，还给假设条件。
- 证据链仍可追溯。
- 不满足因果假设时明确拒绝或降级。

### 7.4 反事实推演

**作用**

回答“如果当时 X 没发生，Y 会怎样”。

**开发架构**

- crate：`causis-causal`。
- API：`POST /api/v1/query/counterfactual`。

**实现方式**

反事实必须基于已知因果图和历史观察。输出 observed world、counterfactual world、changed variables、reasoning path。

**验收标准**

- 反事实结论有前提声明。
- 不支持的反事实不强行回答。
- 审计日志记录查询参数。

### 7.5 根因定位

**作用**

从多个异常信号中定位可能原因。

**开发架构**

- crate：`causis-causal`。
- 组件：`RootCauseAnalyzer`。

**实现方式**

先用图路径、时间先后、质量分数和规则权重计算候选根因排名。P2 再考虑 Shapley 或更复杂方法。

**验收标准**

- 输出候选原因排序。
- 每个原因有证据链。
- 结果区分 confidence 和 explanation。

## 8. Phase 5：平台与生态

### 8.1 Axum API Server

**作用**

让 Causis 从 CLI demo 变成可集成服务。

**开发架构**

- app：`causis-server`。
- crate：`causis-api`。

**实现方式**

提供 evidence、facts、graph、conflicts、trace、explanation、review 等 API。API 层只做请求校验和服务编排，不写业务逻辑。

**验收标准**

- OpenAPI 可生成。
- API 有集成测试。
- 错误响应统一。

### 8.2 Causis Studio

**作用**

提供可视化工作台，展示证据链、图谱、冲突队列和因果解释。

**开发架构**

- app：`causis-studio`。
- 数据来源：Causis API + Cube analytics API。

**实现方式**

先做内部管理界面，不做营销页。核心视图包括：证据浏览、事实表、冲突队列、图谱路径、trace 解释、审计日志。

**验收标准**

- 能打开一个 trace 并查看每步证据。
- 能审核冲突。
- 能查看系统运行指标。

### 8.3 Cube Analytics Layer

**作用**

为 Studio、BI 和 Agent 提供指标语义层。

**开发架构**

- adapter：`causis-adapter-cube`。
- 数据源：DuckDB/Postgres/Trino/Athena/Iceberg SQL 层。

**实现方式**

Cube 不进入 core。Causis 输出 facts、conflicts、resolutions、audit_events、causal_traces 表；Cube 定义 metrics、dimensions 和 pre-aggregations。

**验收标准**

- 能查询冲突趋势。
- 能查询自动裁决率。
- 能查询数据源质量分数。
- BI 工具不直接依赖底层表结构。

### 8.4 MCP Server

**作用**

让 AI Agent 调用 Causis 的可信推理能力。

**开发架构**

- app：`causis-mcp-server`。
- 工具：`causal_trace`、`explain_decision`、`query_evidence`、`list_conflicts`、`get_audit_report`。

**实现方式**

MCP Server 调用 Causis API 或直接调用 core service。工具输入必须结构化，输出必须包含 evidence references，不返回裸结论。

**验收标准**

- Agent 可以查询某个决策的证据链。
- 工具输出包含 trace id。
- 高风险工具有只读限制。

### 8.5 SDK

**作用**

降低外部系统集成成本。

**开发架构**

- Rust SDK：直接使用 core/client。
- Python/TypeScript SDK：通过 API schema 生成或手写薄封装。

**实现方式**

优先提供 query、evidence、trace、explanation、review 相关方法，不暴露内部存储细节。

**验收标准**

- 10 行以内代码跑通 trace 查询。
- SDK 示例和 CI 测试同步。

### 8.6 Marketplace

**作用**

让 Causis 从产品变成生态。

**开发架构**

- 平台功能，不放在 Core。
- 包类型：connector、causal template、rule pack、schema mapping、dashboard。

**实现方式**

先用 Git 仓库和 manifest 文件管理模板。后续再做在线市场。模板必须包含数据契约、规则说明、测试 fixture 和许可证。

**验收标准**

- 官方提供至少 3 个模板。
- 模板可安装、可测试。
- 模板输出一致。

## 11. 第一轮开发建议

第一轮不要写太多基础设施代码。推荐只做这些：

1. `cargo init` workspace。
2. `causis-core`：ID、error、provenance、fact 基础类型。
3. `fixtures/leave-approval`：请假审批样例。
4. `causis-cli demo leave-approval`：先用内存实现跑通。
5. 输出 `explanation.json`。
6. 为每一步写测试。

如果这一步跑通，Causis 就从“好想法”进入“可验证项目”。后面的 RustFS、Iceberg、LadybugDB、Cube、MCP 都会有清晰接入点。
