## 模块六：冲突检测与协调引擎 设计方案

### 1. 模块定位

冲突检测与协调引擎是 Causis **“可信推理”架构的最终裁决层**。在企业真实数据环境中，多源异构数据（不同版本的PDF制度、HR系统快照、OA审批记录）对同一事实给出矛盾结论是常态，而非异常。模块三已在事实层忠实地发现并标记了这些冲突，模块六的核心使命是：**作为冲突的终局裁决者，以透明、可审计、人机协作的方式消解冲突，产出系统最终采信的权威事实版本。**

**核心职责：**

| 职责                       | 说明                                                         |
| :------------------------- | :----------------------------------------------------------- |
| **冲突接收与归并**         | 从模块三的事实表中消费 `conflict_status = detected` 的冲突记录，按 `conflict_group_id` 归并为待协调的冲突组 |
| **多策略自动冲裁**         | 基于可配置的协调策略库（权威性优先/时间优先/多源投票/质量加权），对大部分冲突自动裁决 |
| **AI辅助消解建议**         | 利用轻量级模型或外部LLM API，对中高复杂度冲突生成消解建议及推理依据 |
| **人在回路审核**           | 为自动冲裁置信度不足或高风险的冲突，提供人工审核接口，记录人工裁决结果 |
| **裁决结果回写与反馈闭环** | 将最终裁决写回模块三事实表（`conflict_status = resolved`），并作为高置信度训练信号反馈给模块五的消歧模型，形成持续学习闭环 |

**边界划分：**

| 排除项                     | 归属                         |
| :------------------------- | :--------------------------- |
| 冲突的初始检测与标记       | 模块三（事实层）             |
| 复杂模糊实体的消歧（L3级） | 模块五（实体消歧与关系补全） |
| 因果推理与反事实推演       | 模块七（因果推理引擎）       |
| 知识图谱的存储与查询       | 模块四（知识图谱层）         |

### 2. 核心设计原则

- **裁决透明化**：每一次冲突消解决策都记录完整的裁决理由、采纳与未采纳的证据版本、决策来源（自动/人工），形成可审计的裁决链。
- **策略可编排**：协调策略不是硬编码的分支逻辑，而是可配置、可组合的策略单元，支持按业务场景定制策略链。技术路径参考了 Oracle GoldenGate CDR 的可配置冲突解决架构。
- **渐进式裁决**：对已检测冲突采用分层裁决策略，从轻量确定性规则开始逐步升级，仅在必要时引入资源密集型的AI推理或人工审核，以最小化处理成本。
- **持续进化**：人工裁决的累积作为标注数据，周期性优化自动冲裁策略和AI模型，使裁决能力随系统运行持续提升，形成正向飞轮。
- **纯Rust轻量核心**：核心裁决引擎全Rust实现，AI辅助模块通过ONNX轻量模型或外部LLM API接入，不引入额外的持久化数据库——裁决日志直接写入模块四的 LadybugDB，复用知识图谱的存储与查询能力。

### 3. 裁决策略的分层设计

如前所述，模块三（事实层）负责“发现与标记”，模块六（冲突协调引擎）负责“消解与裁决”。这种职责分离使得两模块可以独立演化——模块三的检测策略升级不影响模块六的裁决逻辑，反之亦然。

### 4. 逐层裁决设计

#### 4.1 第一层：确定性规则自动裁决

本层处理可基于先验规则直接裁决的冲突，零人工介入，毫秒级响应。

| 策略               | 触发条件                   | 裁决逻辑                                 | 信任度 | 适用场景                                 |
| :----------------- | :------------------------- | :--------------------------------------- | :----- | :--------------------------------------- |
| **时间优先 (LWW)** | 非规则类属性冲突           | 采纳最新 `valid_from` 的事实版本         | 0.85   | 时效性强的数据（如员工状态）             |
| **权威优先**       | 冲突源中明确包含高权威源   | 采纳 `source_authority_score` 最高的版本 | 0.9    | HR系统数据优先于手动上传的Excel          |
| **完整性优先**     | 候选值中存在明显完整度差异 | 采纳包含信息更丰富且不矛盾的版本         | 0.8    | 一个版本有完整地址信息，另一个仅有城市名 |
| **人工锁定**       | 曾被人工裁决且上下文未变   | 直接复用历史裁决结果                     | 0.95   | 已确认的决策避免重复审核                 |

**设计要点**：LWW 算法的核心优势在于实现简单、计算开销低，但会丢弃冲突版本的历史。Causis 在此基础上做了关键增强——被丢弃的版本保留在 `alternative_values` 字段中并保持可审计，不丢失任何历史证据。所有决策记录在 `conflict_resolution` 裁决日志表中（存储于 LadybugDB）。

#### 4.2 第二层：质量驱动的多维度加权投票

本层处理无法被确定性规则裁决的冲突。系统引入来源质量权重和值可信度权重，通过多维度加权投票实现自动化裁决，准确率显著优于简单的多数投票。

| 权重维度             | 计算来源                                                     | 说明                                       |
| :------------------- | :----------------------------------------------------------- | :----------------------------------------- |
| **来源权威性权重**   | 模块三事实表的 `quality_score` 中预设的来源权威性子分        | 正规HR系统 > OA系统 > 手动上传的Excel      |
| **数据新鲜度权重**   | 基于 `valid_from` 的时间衰减函数                             | 越近的数据权重越高（半衰期可配置，如90天） |
| **多源一致性权重**   | 多个独立来源提供相同值时，每个来源的权重乘以一致性奖励系数（1.2x） | 三个不同系统同时记录了相同的员工部门信息   |
| **人工裁决复用权重** | 若同一冲突组中的某个候选值曾被人为确认为正确，且相关上下文未发生变化，则该候选获得额外的信任分 | 复用历史上人工裁决的知识，避免重复审核     |

**裁决流程**：
1. 对冲突组的每个候选值，计算多维度加权总分。
2. 得分最高的候选值若满足以下条件之一，自动采纳：（a）最高分与次高分差值 **> 0.2**；或（b）最高分 **> 0.8**。
3. 不满足上述阈值时，升级到第三层。

#### 4.3 第三层：LLM辅助消解建议

本层处理前两层无法自动裁决的复杂冲突（如语义冲突、推理冲突）。引入 LLM 的语义理解能力，生成可解释的消解建议，并征求人工介入。

参考 CRDL（Detect-Then-Resolve）方法，该框架结合冲突检测和LLM来识别真相，在 recall 和 F1-score 上分别取得了 56.4% 和 68.2% 的提升，证明了“先检测、后LLM分辨”在冲突消解场景中的有效性。本模块采纳了该范式的核心洞察：将精确的检测过滤策略与LLM的语义推理能力协同使用，而非依赖于纯端到端的黑盒裁决。

**LLM Prompt 设计策略**：
- **实体链接增强**：注入冲突实体在 LadybugDB 中的邻接子图（直接汇报链、同部门成员等），帮助 LLM 理解实体间的结构化关系。
- **时间线构建**：将冲突的多个版本按时间排序，形成“事实演化时间线”注入 Prompt，辅助 LLM 判断版本演化路径的合理性，减少因信息缺失导致的误判。
- **证据溯源注入**：每个候选值附带 `_source_file` 和 `_source_location`，让 LLM 基于来源可信度和具体证据做出判断，减少幻觉。

**LLM 的输出格式**：
- **决策**：推荐哪一个候选值（或给出合并建议）。
- **置信度**：0-1 的标量，低于阈值的自动升级至第四层人工审核。
- **推理依据**：自然语言解释，引用证据来源和推理逻辑。
- **替代方案**：若存在合理的合并可能（如取最新版本的有效区间并保留历史记录），一并给出合并建议。

**MVP 实现方式**：
- **可选模型配置**：支持轻量本地模型（ONNX格式，零网络依赖）或外部LLM API（如 OpenAI / Claude / DeepSeek），通过配置文件切换。在模型不可用时，系统自动降级为纯规则裁决，不影响核心管线的运行。

#### 4.4 第四层：人在回路人工审核

本层处理前第三层均无法自动裁决的高难度冲突，以及业务关键性极高的审核场景。TCR（Transparent Conflict Resolution）等框架的研究表明，让人工审核者能直接观测和控制冲突处理的过程，可以提升冲突检测的F1值5-18分以上，并减少近30%的误导性错误。这证实了“透明化决策过程”在高冲突场景中的切实价值。

**自动升级条件**：
- 多源投票最高分与次高分差值 **< 0.1**（候选值几乎持平）。
- LLM 给出的置信度 **< 0.7**。
- 冲突涉及关键字段（如“审批人”标识字段），无论策略结果如何，一律标记为需人工复核。
- 冲突源中包含手动上传文件（不可靠性较高），且与其他系统来源存在矛盾的。

**审核流程**：

1. **分级队列管理**：所有待审核冲突按严重等级（高/中）和业务域（HR/财务/法务）自动分级。高优先级冲突（涉及关键审批人、金额阈值等）优先展示，确保核心决策不被阻塞。
2. **完整上下文呈现**：审核界面展示冲突实体的完整知识图谱子图（Cypher 查询 LadybugDB），包含冲突实体的关联节点与边、所有候选值及其来源和证据锚点、自动冲裁策略的建议结果与置信度，以及时间线视图。
3. **审核操作处理**：支持“选择版本 A”、“选择版本 B”、“标记为需补充信息”、“创建合并版本”等操作。审核决策通过 REST API 提交，裁决结果立即回写到模块三的事实表中。
4. **审计信息记录**：每次人工审核记录审核人、决策、理由和审核时间戳，作为裁决日志的一部分持久化存储。

### 5. 裁决日志表设计

所有裁决决策统一写入 LadybugDB（复用模块四的存储层），以节点和边的形式融入知识图谱，便于后续审计查询和模型训练。

**裁决节点 (ResolutionNode)**：

| 字段                    | 类型      | 说明                                                         |
| :---------------------- | :-------- | :----------------------------------------------------------- |
| `resolution_id`         | UUID      | 裁决唯一标识                                                 |
| `conflict_group_id`     | UUID      | 对应事实表中的冲突组ID                                       |
| `resolution_layer`      | Enum      | `rule_based` / `quality_weighted` / `llm_assisted` / `human_in_loop` |
| `strategy`              | String    | 具体裁决策略名称                                             |
| `adopted_value`         | Variant   | 最终采纳的事实值                                             |
| `adopted_source`        | String    | 采纳值的数据来源路径                                         |
| `confidence`            | Float     | 裁决置信度                                                   |
| `reasoning`             | String    | 裁决理由的完整自然语言描述                                   |
| `reviewer`              | String    | 审核人标识（自动裁决则为 `"system"`）                        |
| `reviewed_at`           | Timestamp | 裁决时间                                                     |
| `rejected_alternatives` | JSON      | 未被采纳的候选版本及其淘汰原因                               |

### 6. 反馈闭环与持续学习

人工裁决的累积作为高质量标注数据，周期性回馈到系统中，形成持续进化：

- **反馈路径**：每次人工裁决完成后，以 `(conflict_group_id, adopted_value, rejected_alternatives)` 三元组的形式追加到裁决反馈队列中。
- **模型优化**：当累积的人工裁决样本达到预设阈值（如100条），触发模块五语义匹配模型（BGE/GTE）的增量微调，提升后续相似冲突的自动冲裁准确率，减少未来的人工介入需求。
- **效果评估**：每月生成一次裁决准确率报告，监控从第一层到第四层的逐层裁决命中率分布，观察各层的自动化裁决比率是否随系统的运行而持续提升。

### 7. 与上下游模块的接口契约

| 接口                           | 方向 | 数据内容                                       | 格式                                    | 协议                           |
| :----------------------------- | :--- | :--------------------------------------------- | :-------------------------------------- | :----------------------------- |
| ← 模块三（事实层）             | 输入 | `conflict_status = detected` 的记录集          | Iceberg 表（通过 DuckDB 查询消费）      | DuckDB SQL → Arrow RecordBatch |
| ← 模块四（知识图谱层）         | 输入 | 冲突实体的邻接子图                             | LadybugDB Cypher 查询结果               | LadybugDB FFI                  |
| → 模块三（事实层）             | 输出 | 裁决后的事实值（`conflict_status = resolved`） | Iceberg MERGE 操作（通过 iceberg-rust） | iceberg-rust API               |
| → 模块四（知识图谱层）         | 输出 | 裁决日志节点和边                               | 写入 LadybugDB                          | LadybugDB FFI                  |
| → 模块五（实体消歧与关系补全） | 反馈 | 人工裁决训练样本                               | 文件系统中的 JSONL 文件                 | 模块五定期拉取                 |
| → API 层                       | 输出 | 待审核队列、裁决结果                           | REST API JSON                           | Axum HTTP                      |

**各模块的职责边界说明**：
- **模块六 vs 模块五**：模块六裁决的是“同一实体同一属性的多个**事实版本**谁正确”，模块五解决的是“两个模糊实体记录**是否指向同一物理实体**”。两者处理的数据对象不同，不存在重复裁决。
- **模块六 vs 模块四**：模块六自身不维护独立的数据库实例，所有裁决日志复用 LadybugDB 的存储能力。这延续了 Causis “尽量复用现有基础设施，不引入新数据库”的轻量化原则。
- **模块六 vs 模块三**：模块三的“数据处理管道”只负责在写入事实表之前完成数据清洗、标准化和冲突检测标记，模块六的“裁决管道”独立运行、异步消费模块三标记好的冲突记录。两者以 Iceberg 表为共享契约，互不侵入对方的代码路径。

### 8. 实施路线图（MVP 阶段）

| 步骤   | 任务                                                         | 核心产出                                        | 关键组件                                        |
| :----- | :----------------------------------------------------------- | :---------------------------------------------- | :---------------------------------------------- |
| **S1** | 构建确定性裁决引擎，实现时间优先、权威优先、完整性优先、人工锁定四项裁决策略 | 可处理约60%普通冲突的规则裁决核心               | Rust 自研策略引擎                               |

| **S3** | 接入 LLM 辅助消解建议能力，支持冲突实体子图查询              | 复杂语义冲突的AI辅助裁决可用                    | LadybugDB Cypher + LLM API适配层 + ONNX备用模型 |

### 9. 总结

模块六通过**四层渐进式裁决架构**，将决定性的规则与AI辅助柔性推理相结合，以高度自动化的方式消解Causis中多源数据带来的矛盾。其核心差异化体现在：

- **裁决透明化**：每次决策保留完整的裁决理由和未被采纳的证据版本，裁决过程完全可审计。
- **分层演进**：从毫秒级规则到人在回路的渐进式设计，在效率与质量之间动态平衡，自动裁决比率随使用持续提升。
- **轻量纯Rust**：裁决核心引擎零外部依赖，AI辅助通过ONNX轻量模型或可选的LLM API灵活接入，裁决日志复用知识图谱的存储能力，不引入新数据库。
- **持续学习**：人工裁决信号反馈至消歧模型，使裁决准确率随系统运行而持续进化。
