完全理解。Causis的定位是“因果即服务”的高性能基础设施，核心技术栈必须轻量、高效且可嵌入。用Rust重构模块一是正确的方向——它在系统软件领域已充分验证，能提供零成本抽象、内存安全和卓越的并发性能。

以下是基于Rust技术栈和自研思路，完全重设计的**模块一：多源数据接入与适配**方案。

---

## 核心设计哲学：从“智能感知闭环”到“轻量高性能发动机”

我们将架构精简为三个核心原则：

1.  **无服务依赖（Serverless/Dependency-Free）**：核心引擎编译为单一静态二进制文件，可嵌入任何环境，零运维依赖。
2.  **流批在编译时统一**：用Rust的类型系统和零成本抽象，在编译时区分流/批路径，而非依赖重量级运行时。
3.  **AI推理内联化**：轻量级模型推理直接在进程中完成，不调用外部LLM服务（但保留外部接口作为可选增强）。

## 子模块重设计与Rust实现路径

### 子模块1：高性能协议适配核心

**设计目标**：以最低资源消耗接入数百种协议，且可扩展。

**Rust实现路径**：

-   **自研适配器框架 `Ingestor-Core`**：
    -   使用 `tokio` 异步运行时作为基础IO层，提供极致的连接并发能力。
    -   适配器以 `trait` 形式定义接口，编译为WASM或动态库形式的热插拔插件，参考 `wasmtime` 作为安全的插件运行时。
    -   每个连接器是一个实现了 `SourceConnector` trait 的结构体，包含 `connect()`, `read()`, `get_schema()` 等方法。

-   **替代JDBC/ODBC的轻量方案**：
    -   **PostgreSQL**：`tokio-postgres` 原生异步，支持COPY协议批量抽取和逻辑复制（CDC）。
    -   **MySQL**：`mysql_async`，纯Rust实现，零C绑定依赖。
    -   **SQLite**：`rusqlite`，嵌入式场景首选。
    -   所有数据库连接器统一返回 `Arrow RecordBatch` 流，作为系统内部标准数据交换格式。

-   **智能Schema推断**：
    -   不使用LLM。基于Rust实现轻量级统计引擎，分析前N行数据，推断类型、分布、约束（非空、唯一性）。
    -   嵌入小型规则引擎（如自研的 `matchit` 宏），根据字段命名约定自动映射语义类型。

**可行性验证**：`tokio`、`tokio-postgres`、`mysql_async` 等均已在生产环境大规模验证（AWS、Discord等）。Rust异步生态成熟。

**MVP阶段交付物**：`Ingestor-Core` 框架 + PostgreSQL连接器 + 本地文件系统连接器，编译为单个二进制，内存占用 < 20MB。

### 子模块2：流批一体的轻量数据管道

**设计目标**：零外部依赖实现流批处理，编译时调度。

**Rust实现路径**：

-   **自研两级数据通道**：
    -   批量路径：基于 `Arrow` 的 `RecordBatch` 流，每批处理数千至数万行。使用 `tokio::spawn` 生成并行处理任务。
    -   实时CDC路径：直接消费数据库复制流（如PostgreSQL的`pgoutput`），无需独立的Kafka/消息队列。解析出的变更事件同样转为 `Arrow` 结构，写入内存中的 `chunked_vec` 缓冲区，批量刷入Iceberg。
    -   需要跨服务投递的场景，嵌入 **`NATS`**（Rust客户端 `async-nats`），它是一个极简高性能消息系统，比Kafka轻两个数量级。

-   **背压与流控**：
    -   利用 `tokio` 的 `Semaphore` 和 `mpsc` 通道实现精细化的背压控制。
    -   用Rust的生命周期机制确保内存安全，通过 `bounded` 通道限制在途数据量。

-   **编译时调度器**：
    -   利用Rust的 `const generics` 和 `proc macro`，在编译时生成针对特定数据源的优化代码路径，消除运行时分支预测开销。

**可行性验证**：Arrow DataFusion（Rust原生查询引擎）已证明Arrow格式的流式处理效率。NATS被广泛应用于边缘计算和IoT，成熟可靠。

**MVP阶段交付物**：批量路径完整实现；CDC路径支持PostgreSQL逻辑复制。

### 子模块3：非结构化文档提取器 `DocTor`

**设计目标**：高性能、可嵌入的文档解析引擎，不依赖外部API。

**Rust实现路径**：

-   **自研文档解析管线**：
    -   **PDF**：使用 `pdf-extract` 或 `lopdf` 解析文本和布局坐标。对于扫描版，集成 `leptess`（Tesseract C库的Rust绑定）进行OCR。
    -   **Word (.docx)**：`.docx` 本质是ZIP+XML，使用 `zip` crate解压，`quick-xml` 解析XML，提取段落、表格和样式信息。**完全自研，零外部依赖**。
    -   **Excel (.xlsx)**：`calamine` 纯Rust库，读取数据并保留sheet、行号等元数据。
    -   **PPT (.pptx)**：同上ZIP+XML方法，提取幻灯片顺序和文本/图片位置。

-   **版式重建算法**：
    -   所有文档统一转为内部 `Document` 中间表示（IR），包含：逻辑结构树（chapter→section→paragraph）、表格、图片、每块文本的`BoundingBox`坐标。
    -   使用预训练的布局模型（如LayoutLM的ONNX版本，用 `ort` 加载），在进程中完成版式分类，或将此作为可选增强特性。

-   **结构化抽取**：
    -   提供 `extract_table()`, `extract_fields()` 等API，基于规则+轻量级ML组合，从文档中提取关键字段。

**可行性验证**：`calamine` 已被Apache OpenOffice等使用；`quick-xml` 是Rust生态中最快的XML解析器。整体方案已在若干开源文档系统中得到验证。

**MVP阶段交付物**：`DocTor` 独立库，支持 `.pdf`（文本）、`.docx`、`.xlsx` 三种格式，输出 `Document` IR。

### 子模块4：语义与质量中心 `MetaKeeper`

**设计目标**：轻量级元数据管理与质量监控。

**Rust实现路径**：

-   **数据质量引擎**：
    -   使用 `DataFusion` 作为内嵌SQL引擎，对输入数据执行用户自定义的质量SQL检查。
    -   质量指标（完整性、唯一性、时效性）通过 `DataFusion` UDF实现，计算结果写入 `Iceberg` 的质量元数据表。

-   **语义对齐**：
    -   MVP阶段，语义映射由用户提供的JSON配置文件驱动（如 `field_mappings.json`），`MetaKeeper`在摄入时自动应用。
    -   更智能的对齐（基于嵌入向量）留待P1实现，可嵌入 `ort` 加载轻量级文本嵌入模型。

-   **健康监控**：
    -   用 `metrics` facade crate 暴露Prometheus兼容指标，`tracing` crate 实现分布式追踪。
    -   健康检查端点通过 `axum` 轻量Web框架暴露，无额外Agent。

**可行性验证**：`DataFusion`（作为内嵌SQL引擎）已在InfluxDB 3.0等多款数据库产品中采用；Prometheus指标格式是云原生标准。

**MVP阶段交付物**：`MetaKeeper` 核心，包含基于JSON的Schema映射和基于DataFusion SQL的基本质量检查。

## 整体技术栈总览（MVP）

```
┌─────────────────────────────────────────────────────────────┐
│  Causis 模块一 MVP (Rust)                                  │
├─────────────────────────────────────────────────────────────┤
│  Ingestor-Core (适配器框架 + PostgreSQL + File)               │
│  DocTor       (PDF/DOCX/XLSX → Document IR)                  │
│  Data Channel (Arrow RecordBatch 流 + 批量路径)               │
│  MetaKeeper   (JSON Schema映射 + DataFusion质量SQL)           │
│  Runtime      (tokio + axum + tracing)                       │
├─────────────────────────────────────────────────────────────┤
│  外部依赖     (MinIO, NATS)    # NATS 仅用于资源解耦场景                         │
└─────────────────────────────────────────────────────────────┘
```

**编译产物**：单一二进制文件，可运行在 `glibc`/`musl` 环境下，Docker镜像 < 50MB。

## 与后续模块的接口

-   **→ 模块二（证据湖）**：`Ingestor` 输出的 `Arrow RecordBatch` 直接写入 `Apache Iceberg` 表（通过 `iceberg-rust` 或 `polars-iceberg`），文档IR以 `lance` 格式存入MinIO。
-   **→ 模块五（实体对齐）**：`DocTor` 输出的结构化和溯源元数据，为实体消歧提供输入。
-   **→ 模块七（因果推理）**：`MetaKeeper` 中的质量分数作为推理链路置信度的基础权重。
