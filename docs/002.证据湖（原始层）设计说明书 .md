## 模块二：证据湖（原始层）设计说明书 v2.0

### 1. 功能概述

证据湖（Evidence Lake）是 Causis 的不可篡改数据底座，负责：

- **原始数据接收与安全存储**：以不可变方式保存所有入湖的原始文件（PDF、Word、Excel、PPT）、数据库快照、云端对象。
- **版本化与时间旅行**：对每一次数据变化生成不可逆的版本快照，支持任意历史时刻的数据状态回溯。
- **结构化 SQL 查询支持**：从证据湖中直接对结构化数据和元数据进行 SQL 查询，无需额外 ETL 即可进行探索、过滤和聚合。
- **全链路溯源锚定**：记录每个数据块的来源、处理步骤和在原始文件中的位置，为上层因果推理提供完整的审计线索链。

### 2. 核心设计原则

- **数据即证据，不可篡改**：原始数据一经写入永不删除或覆盖，所有修正是通过追加新版本实现。
- **查询与分析分离**：借助湖仓一体表格式，让同一份数据可被多种轻量引擎分析，无需复制。
- **极致轻量与 Rust 原生**：核心组件均在 Rust 生态内实现，单二进制部署，环境依赖为零。
- **开放标准，避免锁定**：存储层采用 S3 兼容协议，表格式采用 Apache Iceberg，计算层可插拔。

### 3. 架构设计

```
┌─────────────────────────────────────────────────────────────┐
│                    证据湖 (Evidence Lake)                    │
├─────────────────────────────────────────────────────────────┤
│  SQL 查询                        │  DataFusion / Polars /   │
│  (标准SQL, 时间旅行)            │  DuckDB (可插拔)        │
├──────────────────────────────────┼──────────────────────────┤
│  表格式 & 版本控制               │  Apache Iceberg          │
│  (快照, 事务, 演变)             │  (iceberg-rust)          │
├──────────────────────────────────┼──────────────────────────┤
│  对象存储 (S3 兼容)             │  RustFS (替代 MinIO)     │
│  (不可变对象, 生命周期)        │  或 Garage (边缘/测试)  │
└──────────────────────────────────┴──────────────────────────┘
```

工作流程：

1. **模块一** 输出的数据（`Document IR`、`Arrow RecordBatch`、原始文件字节流）通过证据湖统一的写入 API 进入系统。
2. 原始文件（`.pdf`、`.docx` 等）直接存入 RustFS 的 `raw` 存储桶，生成对象路径和哈希值。
3. 结构化数据和元数据被转换为 Apache Iceberg 表，存储在 RustFS 的 `warehouse` 桶中。
4. 上层查询引擎（DataFusion 等）通过 Iceberg 目录接口直接访问这些表，支持标准 SQL 和时间旅行。

### 4. 核心技术选型

#### 4.1 对象存储：RustFS（替代 MinIO）

- **选型原因**：MinIO 转向 AGPLv3 并进入维护模式，对商业部署构成风险。RustFS 采用 Apache 2.0 协议，具有与 MinIO 完全兼容的 S3 API，并且在性能上表现更优（特别是在小文件高并发场景下吞吐量可达到 MinIO 的 2.3 倍）。
- **关键特性**：
  - 完全 S3 兼容，无需修改现有 S3 客户端库（如 `object_store`、`rusoto`）。
  - 体积 < 100 MB，可嵌入部署，内存占用极低。
  - 支持对象版本控制，可直接作为原始文件的不可变存储。
  - 支持静态加密、生命周期管理和 IAM 策略。
- **替代方案**：
  - **Garage**（轻量级边缘存储，AGPLv3）：适合极低资源消耗的边缘或测试部署。
  - **borderless-storage**（MIT 协议，极简内核）：作为自研存储引擎的参考原型，但在生产环境成熟度上不及 RustFS。

#### 4.2 表格式 & 版本控制：Apache Iceberg

- **选型原因**：Iceberg 是云原生湖仓一体的标准，能够提供 ACID 事务、快照隔离和时间旅行，且生态完善、性能优异。其开放标准特性与 Causis 的“避免锁定”原则完全一致。
- **关键特性**：
  - 表结构演化支持完整 DDL（`ADD COLUMN`、`RENAME COLUMN` 等），无需重写数据。
  - 快照基于 DAG 结构，每次写入都生成新快照，支持 `FOR SYSTEM_TIME AS OF` 语法进行时间旅行。
  - 通过 `iceberg-rust` crate 直接集成 Rust 原生操作，无需依赖 JVM。
  - 所有表均可通过 SQL 查询，配置不同的计算引擎（DataFusion、Polaris 等），均能获得一致的结果。
- **在证据湖中的应用**：
  - **文件元数据表**：记录每个原始文件的对象键、哈希、大小、上传时间、源信息等。
  - **溯源关系表**：记录从原始文件到事实抽取的每一步映射，包括源文件、处理工具、抽取结果表等。
  - **结构化事实表**：存放从模块三输出的事实数据，其中每条记录都附带 `_provenance_id` 外键关联至溯源关系表。

#### 4.3 查询引擎：DataFusion / Polars / DuckDB（可插拔）

- **主推内嵌引擎**：**DataFusion**（通过 `iceberg-rust` 集成），它能够直接在 Rust 进程中执行 SQL 查询 Iceberg 表，无网络开销，响应延迟极低。
- **数据处理任务**：**Polars** 是高性能 DataFrame 库，支持直接读取和写入 Iceberg 表，适合数据清洗、聚合和特征转换等任务。
- **嵌入式 BI / 分析**：**DuckDB** 通过扩展可直接查询 Iceberg 表，可非常简便地集成到 Web 服务中，提供即席分析功能。
- 所有引擎均支持标准 SQL 的 `SELECT`、`WHERE`、`GROUP BY`、`JOIN` 及时间旅行等高级功能，用户或上层应用可以根据场景任意选择，证据湖本身对引擎无锁定。

### 5. SQL 查询能力说明

证据湖中的 **所有数据均可通过标准 SQL 查询**，包括：

- **查询文件元数据**：
  ```sql
  SELECT * FROM iceberg.evidence.file_catalog
  WHERE source_type = 'pdf' AND upload_time > '2025-01-01';
  ```
- **查询结构化事实表**：
  ```sql
  SELECT entity_id, attribute, value, _provenance_id
  FROM iceberg.facts.hr_employees
  WHERE entity_id = 'E12345';
  ```
- **时间旅行查询**（查看某一历史版本的数据）：
  ```sql
  SELECT * FROM iceberg.facts.hr_employees
  FOR SYSTEM_TIME AS OF TIMESTAMP '2025-09-15 10:00:00';
  ```
- **联合溯源查询**：
  ```sql
  SELECT e.name, e.department, p.source_file, p.extraction_time
  FROM iceberg.facts.hr_employees e
  JOIN iceberg.evidence.provenance p ON e._provenance_id = p.id
  WHERE e.entity_id = 'E12345';
  ```

这些查询能力使得业务分析师、审计人员甚至前端应用都可以直接利用熟悉的 SQL 工具来探索证据湖，极大降低了使用门槛。

### 6. 溯源与版本控制实现

证据湖通过分层机制实现精确到原始数据片段的溯源：

1. **文件级溯源**：每个文件存储至 RustFS 后，其对象键、内容哈希（SHA-256）和 lakeFS 式分支/提交概念（通过 Iceberg 快照模拟）记录在元数据表中。
2. **记录级溯源**：结构化事实表（Iceberg）中的每一行数据都包含 `_source_file_key` 和 `_source_file_version` 字段，直接引用到 RustFS 中的原始文件。
3. **处理链路溯源**：建立独立的 `provenance` 表，记录从原始文件到最终结论的完整操作链： `文件 → DocTor解析 → 事实抽取 → 知识图谱构建`，每一步都有时间戳和操作者信息。
4. **时间旅行与版本对比**：Iceberg 的快照能力使得任意两个版本的数据可以轻易对比，例如审计某个文档更新前后招聘要求的变化，或回溯某条错误规则的引入时间点。

### 7. 技术可行性验证

- **RustFS 生产可用性**：RustFS 已在多个中大型企业部署替代 MinIO，并通过了 S3 兼容性认证，稳定性和性能经产业界检验。
- **Iceberg + Rust 生态**：`iceberg-rust` 已涵盖完整的 Catalog 和 Table 操作，且与 DataFusion 集成，可实现从写入到 SQL 查询的完整闭环。
- **轻量级部署**：证据湖核心组件（RustFS + Iceberg 元数据服务 + DataFusion）可以编译为单一二进制文件，总计 < 100 MB，运行内存占用 < 150 MB，非常适合边缘、私有化或中小规模部署。
- **开放标准兼容性**：整个架构基于 S3、Iceberg、Arrow 等工业标准，未来可无缝迁移至更大规模的云环境，零迁移成本。

### 8. 实施路径（MVP 阶段）

| 组件     | 技术选型                    | 核心任务                                         | 交付标准                                       |
| :------- | :-------------------------- | :----------------------------------------------- | :--------------------------------------------- |
| 对象存储 | RustFS                      | 部署单节点实例，创建 `raw` 和 `warehouse` 存储桶 | 可接收模块一上传的原始文件和 Arrow RecordBatch |
| 表格式   | Apache Iceberg (iceberg-rs) | 创建文件元数据表、溯源关系表，定义 Schema        | 支持 CRUD 操作和版本快照                       |
| 查询引擎 | DataFusion                  | 集成到 Causis 进程，提供 SQL 接口             | 可以执行上述示例中的 SQL 查询                  |
| 溯源链路 | 自研 Provenance Tracker     | 实现文件写入、事实抽取时自动记录溯源关系         | 能够从任意事实行追溯到原始文件                 |

MVP 交付后，证据湖将具备数据的完整版本控制、标准化 SQL 查询和基本溯源能力，为后续模块（事实层、知识图谱）提供坚实可靠的数据基础。
