# 架构与功能终审报告

审查日期：2026-05-15

本文档是 Causis 进入工程实现前的架构与功能终审。目标不是再扩写愿景，而是判断当前项目是否具备先进性、未来性和可落地性，并给出最终架构裁定。

## 1. 终审结论

Causis 当前方向值得继续推进。

它的先进性不来自单一技术名词，而来自一组组合判断：

- 用因果推理超越传统 RAG 和 GraphRAG。
- 用证据链把 AI 输出变成可审计决策。
- 用冲突协调正面处理企业真实数据的不一致。
- 用 Rust-first 和 adapter-first 保持可嵌入、可替换、可私有部署。
- 用 Cube、MCP、SDK、Studio 和 Marketplace 把核心能力扩展成生态。

最终裁定：

> Causis 应定位为“可信决策基础设施”，以证据、事实、冲突、因果、审计五类核心原语为内核，外接语义层、Agent 工具层和行业模板生态。

这个定位具有未来性，因为未来的企业 AI 不会只需要“生成答案”，而会需要“证明答案为什么可信、是否合规、谁负责、能否复现”。

## 2. 架构先进性审查

### 2.1 已具备的先进性

| 能力 | 评价 | 原因 |
| --- | --- | --- |
| 因果优先 | 先进 | 从相关性检索升级到因果链、干预、反事实和根因 |
| 证据优先 | 先进 | 每个结论绑定原始文件、位置、版本和处理链 |
| 冲突原生 | 先进 | 不假装数据一致，而是显式发现、保留、裁决 |
| 时态建模 | 先进 | 支持规则、事实和关系随时间变化 |
| 审计内建 | 先进 | 审计不是外部日志，而是架构主路径 |
| Adapter-first | 先进且务实 | 保护 Core，不被 RustFS、Iceberg、LadybugDB、Cube 锁死 |
| AI 可降级 | 先进且安全 | LLM 只增强，不成为不可审计的最终裁判 |
| MCP/Agent 友好 | 有未来性 | 可以成为 AI Agent 的可信工具层 |
| Cube 语义层 | 有未来性 | 让 Causis 的运行指标、治理指标、审计指标可被 BI 和 Agent 消费 |
| Marketplace | 有生态潜力 | 把行业知识、规则包、连接器和模板变成网络效应 |

### 2.2 需要坚持的架构原则

以下原则不应在实现中妥协：

1. Core 不直接依赖具体基础设施。
2. 事实层只标记冲突，不做最终裁决。
3. 冲突裁决必须保留 rejected alternatives。
4. 图谱边、规则节点、事实记录都必须带 provenance。
5. LLM 不做最终形式化推理。
6. P0 先做本地闭环，不依赖外部服务。
7. 每个阶段都必须有可运行 demo。
8. 每个推理结果都必须可以生成解释 JSON。

这些原则是 Causis 区别于普通 AI 应用的架构骨架。

## 3. 功能先进性审查

### 3.1 当前功能列表是否足够

当前功能列表已经覆盖可信决策系统的完整闭环：

```text
ingest -> evidence -> facts -> conflicts -> graph -> causal trace -> explanation -> audit
```

这个闭环是合理的。它比单纯的“文档问答”更完整，也比纯数据库或纯图谱系统更面向决策。

### 3.2 必须保留的 P0 功能

P0 不宜再削减以下功能：

| 功能 | 为什么必须保留 |
| --- | --- |
| 核心 ID 体系 | 没有稳定 ID 就没有审计链 |
| Provenance schema | 没有溯源就不是 Causis |
| Local Evidence Store | 支撑本地可跑 demo |
| Fact model | 所有后续图谱和因果推理的基础 |
| Conflict detector | 体现“冲突是资产”理念 |
| In-memory graph | 先验证路径和规则，不等 LadybugDB |
| Rule resolution | 没有裁决就没有可信事实版本 |
| Causal trace | MVP 核心演示价值 |
| Explanation JSON | 给 API、Studio、审计报告打基础 |
| CLI demo | 保证项目不是只有文档 |

### 3.3 可以后移的功能

以下功能有未来性，但不应进入 P0：

| 功能 | 建议阶段 | 原因 |
| --- | --- | --- |
| RustFS 真对象存储 | P1 | P0 本地 adapter 足够验证 |
| Iceberg 真快照表 | P1 | 先用领域接口和 DuckDB/Postgres 过渡 |
| LadybugDB 真实图数据库 | P1 | 先用 in-memory graph 验证模型 |
| Cube 语义层 | P1/P2 | 分析和 BI 层，不是核心闭环 |
| Causis Studio | P2 | CLI 和 API 先跑通 |
| MCP Server | P2 | Core 稳定后再暴露给 Agent |
| 干预和反事实 | P2 | 需要稳定 causal graph |
| Marketplace | P2/P3 | 生态必须建立在可用 Core 之上 |

## 4. 需要补强的未来性

当前规划已经先进，但如果希望项目更有未来性，建议补强以下七个方向。

### 4.1 Causal Contract

未来企业会需要“因果契约”，也就是对每条因果边声明：

- 适用范围。
- 假设条件。
- 证据等级。
- 置信度来源。
- 可干预性。
- 禁止使用的敏感变量。
- 失效条件。

建议新增概念：

```text
CausalContract = causal_edge + assumptions + evidence_level + policy_constraints + validity_window
```

这会让 Causis 的因果图从“关系图”升级成“可治理的因果合约网络”。

### 4.2 Evidence Grade

不是所有证据质量都一样。未来应为证据引入等级：

| 等级 | 含义 |
| --- | --- |
| E0 | 未验证来源 |
| E1 | 单一来源证据 |
| E2 | 多源一致证据 |
| E3 | 权威系统证据 |
| E4 | 人工审核确认 |
| E5 | 合规/监管认可证据 |

这样 Causis 可以回答“不仅有什么证据，还有证据有多强”。

### 4.3 Policy-as-Code

冲突裁决、权限控制、敏感变量约束、审计要求都应逐步进入 policy-as-code。

P0 不需要完整策略引擎，但数据模型应预留：

- policy id。
- policy version。
- policy decision。
- policy explanation。
- policy provenance。

这会让 Causis 更适合金融、医疗、政务等合规场景。

### 4.4 Agent Tool Safety

MCP Server 不只是“让 Agent 能调用 Causis”，还要保证 Agent 调用是安全的。

建议未来 MCP 工具分级：

| 等级 | 工具类型 | 示例 |
| --- | --- | --- |
| Read-only | 只读查询 | 查证据、查 trace、查审计 |
| Advisory | 生成建议 | 冲突建议、根因候选 |
| Simulative | 模拟推演 | 干预、反事实 |
| Mutating | 改变系统状态 | 人工裁决、规则更新 |

P0/P1 只做 read-only 和 advisory。mutating tool 必须有权限和人工确认。

### 4.5 Semantic + Causal 双层模型

Cube 的加入说明 Causis 需要两种语义：

- Semantic Layer：指标、维度、实体口径、分析消费。
- Causal Layer：因果边、干预、反事实、根因。

未来 Causis 的先进形态应是：

```text
semantic layer answers: what is measured?
causal layer answers: why did it happen and what if we intervene?
evidence layer answers: why should we trust it?
```

这三层合起来，会比普通 BI、RAG、图数据库更有未来性。

### 4.6 Benchmark First

如果想成为好项目，Causis 需要自己的评测体系。建议未来建立 Causis Benchmark：

- 证据链完整率。
- 冲突保留率。
- 冲突裁决准确率。
- 因果 trace 可复现率。
- 反事实查询一致性。
- 审计报告可读性。
- 单节点资源占用。
- Agent 工具调用安全性。

这能避免项目只讲概念，无法证明自己先进。

### 4.7 Blueprints 生态

Marketplace 不应一开始做成“大市场”，而应先做 Causal Blueprints：

- `leave-approval-blueprint`
- `expense-compliance-blueprint`
- `contract-conflict-blueprint`
- `credit-risk-explanation-blueprint`
- `manufacturing-root-cause-blueprint`

每个 Blueprint 应包含：

- 数据 schema。
- 规则图。
- 因果图。
- 示例证据。
- 冲突样例。
- 测试 fixture。
- 解释模板。

Blueprint 是 Causis 从工具走向生态的关键抓手。

## 5. 风险终审

### 5.1 最大风险：范围过大

Causis 的想法很大。最大的工程风险不是技术不先进，而是太先进、太完整，导致迟迟不能跑起来。

应对：

- 第一阶段只做请假审批 demo。
- P0 不接真实 RustFS、Iceberg、LadybugDB、Cube。
- P0 不做 UI。
- P0 不做反事实。
- P0 必须一条命令跑通。

### 5.2 第二风险：因果能力过度承诺

因果推理不能被写成“万能解释器”。很多业务场景只有规则链或相关性，不能声称有严格因果。

应对：

- 区分 causal trace、rule trace、evidence trace。
- 输出假设和限制。
- 给每条因果边添加 evidence grade 和 assumptions。
- 对无法支持的因果查询明确拒绝。

### 5.3 第三风险：技术选型过早锁死

RustFS、Iceberg、LadybugDB、Cube 都是合理方向，但如果 Core 直接依赖它们，项目会变重。

应对：

- trait-first。
- local adapter-first。
- ADR 记录技术决策。
- P1 再接真实基础设施。

### 5.4 第四风险：先进性只停留在概念

如果没有 benchmark、fixture、demo 和自动测试，先进性无法证明。

应对：

- 每个核心功能都配 fixture。
- 每个 trace 都可复现。
- 每个解释都可 JSON 输出。
- 每个重要组件都有验收标准。

## 6. 最终推荐架构

终审后推荐架构如下：

```text
┌──────────────────────────────────────────────────────────┐
│ Causis Ecosystem                                          │
│ Blueprints / Marketplace / SDK / MCP Tools                │
├──────────────────────────────────────────────────────────┤
│ Causis Experience                                         │
│ Studio / BI / Cube Semantic Layer / Reports               │
├──────────────────────────────────────────────────────────┤
│ Causis API                                                │
│ REST / OpenAPI / Query Orchestration / Review API         │
├──────────────────────────────────────────────────────────┤
│ Causis Core                                               │
│ Evidence / Facts / Conflicts / Graph / Causal / Audit     │
├──────────────────────────────────────────────────────────┤
│ Causis Adapters                                           │
│ Local / DuckDB / Postgres / RustFS / Iceberg / LadybugDB  │
├──────────────────────────────────────────────────────────┤
│ External Intelligence                                     │
│ LLM API / ONNX / Candle / Optional Model Providers        │
└──────────────────────────────────────────────────────────┘
```

核心裁定：

- `Causis Core` 是产品心脏。
- `Adapters` 是技术可替换性的保障。
- `Experience` 是产品可用性的保障。
- `Ecosystem` 是长期未来性的保障。
- `External Intelligence` 只能增强，不能主宰。

## 8. 是否具备未来性

终审判断：具备，但必须靠工程纪律保护。

Causis 的未来性来自五个趋势交汇：

1. 企业 AI 从问答走向决策。
2. AI Agent 需要可信工具，而不是只需要更多提示词。
3. 合规要求会推动可解释、可追溯、可复现。
4. 数据冲突会成为企业 AI 的常态问题。
5. 因果推理会成为高价值决策的重要基础能力。

Causis 正好站在这些趋势的交叉点。

但要成为好项目，必须把未来性落到最小可运行事实：

> 第一个版本只要能完整证明“一个决策从证据到事实、从事实到图谱、从图谱到推理、从推理到解释、从解释到审计”这条链是可运行的，项目就站住了。

## 10. 终审裁定

Causis 当前架构和功能规划通过终审。

通过条件：

- 不扩大 P0 范围。
- 不让 LLM 进入最终推理裁决。
- 不让具体基础设施污染 Core。
- 不牺牲 provenance。
- 不跳过 CLI demo。
- 不在没有 benchmark 和 fixture 的情况下宣称能力成熟。

如果按这个纪律推进，Causis 有机会成为一个真正先进、面向未来、同时又能落地的可信决策基础设施项目。
