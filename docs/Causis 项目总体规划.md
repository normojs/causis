# Causis 项目总体规划

本文档是 Causis 的顶层规划文档，用于统一愿景、架构、技术路线、功能列表、应用场景和阶段目标。更细的模块设计继续保留在 `docs/001` 到 `docs/008` 中；本文负责回答一个更根本的问题：Causis 要成为一个什么样的好项目。

## 1. 项目判断

Causis 应该被定义为一类新的可信决策基础设施，而不是一个普通知识库、RAG 应用、低代码平台或行业解决方案。

它的核心命题是：

> 当 AI 开始参与高价值决策时，系统不仅要回答“答案是什么”，还要回答“因果链是什么、证据在哪里、冲突如何处理、责任如何追溯”。

因此，Causis 的长期价值不在于比现有 RAG 多检索几段文本，而在于把企业决策需要的四种能力内建为平台原语：

- 因果原语：原因、结果、干预、反事实、根因。
- 证据原语：来源、版本、位置、哈希、处理链。
- 冲突原语：候选事实、冲突组、裁决策略、 rejected alternatives。
- 审计原语：推理轨迹、人工介入、合规约束、报告输出。

这四类原语一旦建立，Causis 就不再只是“一个工具”，而是可以成为 AI Agent、企业应用、行业 ISV 和内部数据平台的可信推理底座。

## 2. 产品定位

### 2.1 一句话定位

Causis 是一个以因果推理为核心、内建全链路证据溯源和冲突协调能力的 AI 原生知识引擎。

### 2.2 目标用户

| 用户 | 他们需要什么 | Causis 提供什么 |
| --- | --- | --- |
| 开发者 | 快速把可信推理能力嵌入应用 | API、SDK、CLI、MCP Server、模板 |
| ISV 和系统集成商 | 给客户交付可解释、可审计的行业应用 | 因果引擎、证据链、冲突协调、私有部署 |
| 企业 IT 和数据团队 | 统一治理多源数据和业务规则 | 证据湖、事实层、知识图谱、审计日志 |
| 审计和合规团队 | 判断 AI 决策是否有依据 | 证据锚点、推理路径、规则版本、报告 |
| AI Agent 平台 | 给 Agent 提供可靠的决策工具 | 因果查询、干预分析、可信解释、工具接口 |

### 2.3 不做什么

- 不做面向单一行业的封闭应用。
- 不做只服务问答的普通 RAG 平台。
- 不把 LLM 作为最终推理裁决者。
- 不以“清洗掉冲突”为目标，而是保留冲突、解释冲突、裁决冲突。
- 不以复杂分布式系统作为起点，先保证单节点、可嵌入、可理解、可验证。

## 3. 先进理念

### 3.1 从检索增强到因果增强

GraphRAG 的价值在于帮助模型找到相关知识。Causis 的价值在于帮助系统计算可信决策。

Causis 不应停留在“图谱增强检索”，而要进入“因果增强决策”：

- 检索回答：哪些材料和问题相关？
- 图谱回答：实体之间有什么关系？
- 因果回答：哪个变量导致了结果？如果干预会怎样？这个结论凭什么成立？
- 审计回答：当时用了哪些证据、规则和版本？谁裁决了冲突？

### 3.2 证据优先，而不是答案优先

企业决策真正的成本不是生成答案，而是证明答案可被信任。

Causis 的所有模块都应围绕“证据可追溯”设计：

- 原始文件不覆盖，只追加版本。
- 事实不脱离来源存在。
- 图谱边和属性必须带来源。
- 冲突裁决必须保留未采纳版本。
- 推理输出必须能回溯到证据锚点。

### 3.3 冲突是资产，不是噪声

企业真实数据天然冲突：制度文件版本不同、系统记录滞后、人工表格错误、规则例外存在。传统系统常把冲突视为需要消灭的脏数据；Causis 应把冲突视为高价值治理信号。

因此，事实层只负责发现和标记冲突，冲突协调层负责透明裁决。这种职责分离是 Causis 的关键架构原则。

### 3.4 LLM 参与构建，不替代推理

LLM 适合做抽取、语义判别、解释润色和低置信度辅助建议，但 Causis 的运行时因果推理应尽量由形式化模型完成。

推荐边界：

- 构建阶段：LLM 可用于文档解析、因果陈述抽取、语义消歧、解释生成。
- 推理阶段：因果查询、干预、反事实、根因定位由形式化因果图和规则执行。
- 审计阶段：LLM 只能润色，不能改写推理事实。

### 3.5 小内核，大生态

Causis Core 应保持小而硬：

- 证据、事实、图谱、因果、审计这些内核能力必须稳定。
- 连接器、行业模板、模型、可视化和场景应用应通过插件和市场扩展。

长期看，Causis 的护城河不是某个单点算法，而是围绕可信决策形成的协议、模型、插件和生态。

### 4.2 数据流

```text
source data
  -> ingest and parse
  -> immutable evidence object
  -> provenance event
  -> normalized fact
  -> conflict marker
  -> graph node / graph edge
  -> causal graph
  -> causal query trace
  -> evidence-backed explanation
  -> audit record / report
```

### 4.3 控制面与数据面

Causis 应从一开始区分控制面和数据面：

| 平面 | 内容 | 目标 |
| --- | --- | --- |
| 数据面 | 原始文件、事实表、图谱、因果边、审计日志 | 高吞吐、可追溯、可版本化 |
| 控制面 | 连接器配置、规则策略、裁决流程、权限、任务调度 | 可配置、可审计、可治理 |
| 推理面 | 因果查询、干预、反事实、根因定位 | 正确性、可解释、可复现 |
| 生态面 | 模板、插件、SDK、MCP、应用市场 | 可扩展、可分发、可复用 |

## 5. 技术方案

### 5.1 技术原则

- Rust 优先：核心系统、内存安全、嵌入式部署、低运维。
- 开放格式优先：Arrow、Iceberg、S3 API、JSON、Cypher/SQL。
- 可替换适配器：对存储、图数据库、LLM、文档解析和因果引擎做 trait 抽象。
- 先单节点闭环，再分布式扩展。
- 先确定性逻辑，再智能增强。
- 所有 AI 能力必须可降级。

### 5.2 推荐技术栈

| 层级 | 技术 | 用途 | 备注 |
| --- | --- | --- | --- |
| API | Axum, tower, utoipa | REST API、OpenAPI、服务中间件 | MVP 即可采用 |
| CLI | clap | 本地管理、导入、测试场景 | MVP 必备 |
| 异步运行时 | tokio | IO、任务调度、连接器 | MVP 必备 |
| 数据交换 | Arrow | 模块间列式数据交换 | MVP 必备 |
| 事实处理 | Polars, DataFusion, DuckDB | 清洗、SQL、质量检查 | 分场景使用 |
| 表格式 | Apache Iceberg | 快照、版本、事实表 | 先定义接口，再接实现 |
| 对象存储 | RustFS / S3-compatible storage | 原始证据和数据湖底座 | 可先本地文件模拟 |
| 图数据库 | LadybugDB | 属性图、时态边、向量检索 | 先以 GraphStore trait 封装 |
| 文档解析 | DocTor, calamine, quick-xml, lopdf | PDF/DOCX/XLSX/PPTX 解析 | 自研管线，逐步增强 |
| 因果推理 | DeepCausality, why-rs, CIfly | 干预、反事实、因果发现、图加速 | MVP 先做因果链追溯 |
| 模型推理 | Candle, ONNX | 本地模型、嵌入、轻量判别 | P1 增强 |
| LLM API | reqwest + provider abstraction | 抽取、裁决建议、解释润色 | 可选增强 |
| 可观测性 | tracing, metrics, Prometheus | 日志、指标、追踪 | MVP 应内建 |
| 插件 | WASM / dynamic connector trait | 连接器与场景扩展 | P1/P2 |

### 5.3 Rust Workspace 建议

初始阶段不要一次性铺开全部复杂 crate，建议按稳定边界拆分：

```text
crates/
  causis-core        # ID、错误、配置、领域类型
  causis-ingest      # 数据接入和文档解析 facade
  causis-evidence    # 证据湖、对象元数据、溯源事件
  causis-facts       # 事实模型、清洗、质量评分、冲突标记
  causis-graph       # 图模型、GraphStore trait、时态关系
  causis-causal      # 因果查询、trace/intervention 抽象
  causis-explain     # 证据链、解释 JSON、报告模型
  causis-api         # Axum API
  causis-cli         # CLI 和 demo runner
```

第一阶段可以先合并成更少 crate，但领域边界应保持清晰，避免后续重构痛苦。

### 5.4 关键接口抽象

| Trait | 责任 |
| --- | --- |
| `SourceConnector` | 从数据库、文件、对象存储或流中读取数据 |
| `DocumentParser` | 将文档解析为统一 Document IR |
| `EvidenceStore` | 写入原始证据并返回对象版本和哈希 |
| `ProvenanceTracker` | 记录处理链路和来源锚点 |
| `FactWriter` | 写入标准事实、质量分数和冲突标记 |
| `GraphStore` | 写入和查询实体、关系、规则和因果边 |
| `ResolutionEngine` | 对冲突组执行策略链裁决 |
| `CausalEngine` | 执行因果追溯、干预、反事实和根因定位 |
| `ExplanationBuilder` | 从 trace 构建证据链和解释输出 |

这些接口是 Causis 的工程骨架。具体库可以换，但接口不应频繁变。

## 7. 应用场景

### 7.1 企业流程决策

典型问题：

- 员工请假应该由谁审批？
- 某笔报销是否符合当时有效制度？
- 某个审批链为什么发生了流转？

价值：

- 决策规则可追溯。
- 审批依据可解释。
- 制度版本和人员关系随时间变化可回放。

这是 MVP 首选场景，因为业务边界清晰、数据类型有限、价值容易演示。

### 7.2 金融合规与风控

典型问题：

- 某个授信拒绝决策的因果依据是什么？
- 某个风险信号来自哪些交易、合同和规则？
- 如果某个变量变化，风险评级是否会改变？

价值：

- 支持合规审计。
- 降低黑盒模型风险。
- 为风控系统提供可解释推理底座。

### 7.3 法律和合同证据智能

典型问题：

- 某条合同义务由哪些条款、补充协议和历史版本共同决定？
- 两份合同条款冲突时应采信哪个版本？
- 某个判例推理链是否适用于当前案件？

价值：

- 文件级和条款级证据锚定。
- 多版本条款比较。
- 可审计的规则推理。

### 7.4 医疗和生命科学

典型问题：

- 某个诊断建议依赖哪些病历、检查结果和指南？
- 某个治疗方案的因果依据是什么？
- 药物重定位候选关系来自哪些文献和实验数据？

价值：

- 证据链对监管和临床解释非常重要。
- 因果推理比相关性检索更适合高风险决策。
- 需要严格区分“建议”“证据”“结论”。

### 7.5 制造与供应链根因分析

典型问题：

- 某次质量异常的根因来自设备、原料、人员还是环境？
- 如果提前更换某零件，故障是否可以避免？
- 哪些供应商变化导致了交付风险？

价值：

- 时序数据、图谱关系和因果推理天然结合。
- 根因定位有明确业务价值。
- 适合从单工厂或单产线 MVP 开始。

### 7.6 AI Agent 可信工具层

典型问题：

- Agent 要做高风险动作前，能否先查因果依据？
- Agent 的建议能否生成可审计报告？
- Agent 调用了哪些证据和规则？

价值：

- Causis 可以成为 Agent 的“可信决策工具”。
- MCP Server 是自然接口。
- 与现有 Agent 框架形成互补，而不是竞争。

## 9. 技术路线图

### 9.1 第 0 阶段：项目固化

目标：把当前文档项目变成可执行工程。

交付：

- Rust workspace。
- 基础 crate。
- GitHub Actions。
- 许可证。
- 贡献指南。
- 示例数据目录。
- 第一个端到端测试规范。

### 9.2 第 1 阶段：MVP 纵向闭环

目标：跑通请假审批或合规报销场景。

交付：

- 本地证据登记。
- 事实模型。
- 小型图谱。
- 冲突标记和规则裁决。
- 因果链 trace。
- 解释 JSON。
- CLI demo。

成功标准：

- 一条命令可复现完整推理链。
- 每个结论都有证据锚点。
- 冲突不会被丢弃。
- 输出可以被审计人员理解。

### 9.3 第 2 阶段：真实数据接入

目标：从样例进入真实企业数据。

交付：

- PDF/DOCX/XLSX 解析。
- Postgres 连接器。
- Iceberg 写入。
- LadybugDB 适配。
- REST API。
- 基础 UI。

成功标准：

- 能处理真实制度文件和人员数据。
- 能追溯原始文档位置。
- 能展示冲突组和裁决结果。

### 9.4 第 3 阶段：因果能力增强

目标：从因果链追溯进入干预和反事实。

交付：

- DeepCausality 集成。
- why-rs 离线发现实验。
- CIfly 可达性加速实验。
- 干预查询。
- 基础反事实查询。

成功标准：

- 能回答“如果 X 不发生，Y 是否会改变”。
- 输出包含假设、路径、证据和限制条件。

### 9.5 第 4 阶段：平台化和生态

目标：让 Causis 可被其他系统稳定集成。

交付：

- SDK。
- MCP Server。
- Studio。
- 模板系统。
- Marketplace MVP。
- 企业版权限与审计增强。

成功标准：

- 外部开发者能在一天内跑通示例。
- ISV 能基于模板构建行业应用。
- 企业能生成标准审计报告。

## 10. 研发原则

### 10.1 工程原则

- 每个模块必须有明确输入、输出和所有权。
- 每个决策对象必须有 ID。
- 每个派生对象必须可追溯。
- 每个非确定性组件必须可降级。
- 每个关键流程必须可通过 fixture 复现。
- 每个技术选型必须能解释替换成本。

### 10.2 文档原则

- README 面向外部读者。
- 总体规划面向项目共识。
- 模块设计面向实现细节。
- ADR 面向重大技术决策。
- 示例场景面向演示和测试。

### 10.3 AI 使用原则

- LLM 输出不能成为不可审计事实。
- LLM 推理建议必须带来源上下文。
- LLM 参与的步骤必须记录模型、提示、输入摘要和输出。
- 高风险结论必须由规则、因果图或人工裁决确认。

## 11. 项目质量标准

一个“好项目”的标准不是文档宏大，而是能持续变成可验证事实。

Causis 的质量标准应包括：

- 可解释：用户能看懂为什么。
- 可追溯：审计员能找到证据。
- 可复现：工程师能重跑流程。
- 可替换：关键外部组件不绑死。
- 可降级：AI 不可用时核心流程仍可运行。
- 可演进：单节点 MVP 能自然升级为平台。
- 可生态：外部开发者能贡献模板和插件。

## 13. 关键风险

| 风险 | 表现 | 应对 |
| --- | --- | --- |
| 概念过大 | 长期停留在规划文档 | 用一个场景纵向闭环约束范围 |
| 技术过散 | 每层都引入复杂组件 | 先 trait + 本地实现，再接真实基础设施 |
| 因果过度承诺 | 把相关性包装成因果 | 明确证据等级、假设条件和限制 |
| LLM 幻觉 | 生成不可验证结论 | LLM 只辅助，不做最终形式化推理 |
| 生态冷启动 | 没有人贡献模板 | 官方先提供高质量行业模板 |
| 企业销售慢 | POC 周期长 | 先做审计价值强、规则明确的场景 |

## 14. 结论

Causis 要成为一个好项目，关键不是“堆先进技术”，而是形成一个清晰、可验证、可持续演进的系统思想：

- 用因果推理超越检索增强。
- 用证据链解决信任问题。
- 用冲突协调面对真实企业数据。
- 用 Rust 和开放格式保证可嵌入、可交付、可替换。
- 用模板、SDK、MCP 和市场形成生态。

如果第一阶段能把“一个决策从原始文件到因果解释再到审计报告”完整跑通，Causis 就有机会从一个想法变成真正有生命力的可信决策基础设施。
